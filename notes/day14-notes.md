# Day 14 Notes — Authentication & Rate Limiting

## Reflection Questions

### 1. SHA-256 Hashes in api_keys.json — What If an Attacker Obtains This File?

If an attacker obtains `api_keys.json`, they get only SHA-256 hashes — not the original keys. SHA-256 is a one-way function: you can compute a hash from a key, but you cannot reverse a hash back to the original key. The attacker cannot use the hashes to make API calls because the middleware validates by hashing the incoming key and comparing it to the stored hash. Without the original plaintext key, the hashes are useless for authentication. The attacker would need to brute-force or rainbow-table attack the hashes, which is computationally infeasible for high-entropy keys like those generated by `secrets.token_urlsafe(32)`.

### 2. Rate Limit Counters on Server Restart — Problem or Reasonable Behavior?

When the server restarts, all in-memory rate limit counters are lost. Every client starts with a fresh slate. This is actually reasonable behavior for a rate limiter: the purpose of rate limiting is to prevent abuse during normal operation, not to persist punishment across restarts. A server restart is an infrequent event, and resetting counters means legitimate users are not penalized for maintenance. In a multi-server production setup, you would use Redis to share counters across instances, but for a single-server internship project, in-memory storage is perfectly acceptable and simpler to implement.

### 3. ADMIN_SECRET from Environment Variable — Why Better Than Hard-Coding?

Hard-coding secrets in source code means anyone with repository access (collaborators, version history, leaked code) can see the secret. Environment variables keep secrets out of the codebase entirely — they are configured per-deployment and never committed to git. To make it even more secure, you could: use a secrets manager (AWS Secrets Manager, HashiCorp Vault), add rate limiting on the admin endpoint itself, require the admin secret to be rotated periodically, or add IP whitelisting so only known admin IPs can access `/admin/keys`.

### 4. Friendly Rate Limit Message in the Browser UI

Instead of showing raw JSON 429 errors, the `sendQuestion()` function in `chat.html` should detect a 429 status code and display a user-friendly message:

```javascript
if (response.status === 429) {
  const data = await response.json();
  const wait = data.retry_after || 60;
  botBubble.textContent = 'You have sent too many messages. Please wait ' + wait + ' seconds.';
  cursor.remove();
  sendBtn.disabled = false;
  return;
}
```

This catches the 429 before attempting to parse SSE streams, shows a clear message with the wait time, and re-enables the input so the user can try again after waiting.

## Key Takeaways

- API key auth is the minimum access control layer every public API must have
- Sliding window rate limiting is fairer and more abuse-resistant than fixed window
- Middleware is the correct place to enforce cross-cutting concerns like auth and rate limiting
- Public paths (/health, /docs) must remain accessible without authentication for monitoring
